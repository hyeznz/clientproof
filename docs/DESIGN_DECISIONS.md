# 디자인 결정의 이유

"왜 이렇게 만들었는가?"를 기록. 나중에 "원래대로 돌릴까?" 고민할 때 참고.

## 1. 폰트: KBL Jump Condensed (히어로)

**선택**: KBLJumpCondensed (EB weight)  
**이유**: 
- 메인 타이틀이 3줄 카피인데 한 줄이 최대 12-14자로 길어짐
- Extended나 Regular는 자폭이 넓어 좁은 모바일에서 오버플로우
- Condensed는 세로로 눌린 형태라 가로 폭 ~60% 절감

**이전 시도 & 실패**: 
- 처음엔 KBLJumpExtended 사용 → 너무 길어 모바일 잘림
- v04w03에서 Condensed로 전환하면서 해결

## 2. 카드 제목 배치: DESC zip (local swap 폐기)

**선택**: DESC zip + promote 2단계 알고리즘  
**이유**:
- 초기엔 local swap 방식 — 원본 순서로 시작, 안 맞는 포지션만 인접 swap
- 문제: 2줄짜리 짧은 후기가 부족한 탭에서는 swap 대상이 없어 실패
  → 4줄 제목이 xs에 갇히는 현상
- DESC zip: 요구 내림차순 × 포지션 내림차순 → 구조적으로 보장
- promote: 그래도 안 맞으면 카드 사이즈 승격 (벽돌 패턴 살짝 변하지만 잘림보단 우선)

**이전 시도 & 실패**:
- v04w21 local swap → 재이사맛집 탭에서 id:45 클리핑 발생
- v04w22에서 DESC zip 전환, 안정화

## 3. 글로벌 단일 FS 후 CSS clamp 환원

**선택**: CSS `clamp(14px, 6.8vw, 26px)` (JS 덮어쓰기 없음)  
**이유**:
- "같은 FS 여러 탭 유지" 공식 때문에 JS로 전역 minFS 계산하는 방식 시도
- 문제: 하나의 까다로운 카드(예: 4줄 제목 xs) 때문에 전체가 작아짐
  → 사용자 "v04w10 사이즈가 제일 좋았다" 피드백
- 해결: JS 덮어쓰기 포기 → CSS clamp으로 기본값 유지
- 여전히 큰 제목이 작은 카드로 가면 클리핑 위험 있지만, 
  rearrangement가 이를 거의 방지

**이전 시도 & 실패**:
- v04w13: 전역 minFS → 너무 작음
- v04w14: 사이즈 클래스별 FS → 공식 ① 위반 (같은 FS 원칙)
- v04w15~16: MULTIPLIER 시도 → 상황 복잡화
- v04w17: CSS 고정 26/36으로 환원 → 사용자 만족
- v04w18: clamp로 narrow 뷰포트 대응
- v04w19~21: 세부 보강

## 4. .kw-line 자연 wrap 허용 (nowrap → normal)

**선택**: `white-space: normal; word-break: keep-all;`  
**이유**:
- nowrap + overflow:hidden 조합은 좁은 화면에서 오른쪽 글자 잘림
- 자연 wrap 허용하면 한 "의도된 줄"이 2줄로 접히며 대응
- titleBreaks는 "권장 기본 레이아웃", 카드 폭이 좁으면 유동적으로 추가 wrap

**이전 시도 & 실패**:
- nowrap 고정 → 모바일 좁은 뷰포트에서 텍스트 잘림
- JS로 FS 축소해서 대응 → 글자 너무 작아짐
- 결국 자연 wrap 폴백이 가장 실용적

---

## 5. `max-width: 100%` 제거

**선택**: .kw-line에서 `max-width` 삭제  
**이유**:
- `max-width: 100%` + `white-space: nowrap` → 요소 박스가 부모 폭으로 캡핑됨
- `offsetWidth` 측정 시 "부모 폭" 반환되어 JS가 "텍스트 길이 OK"로 오판
- 실제로는 내부 텍스트가 오버플로우 상태
- 제거 후 `offsetWidth`가 실제 콘텐츠 폭 반환 → 정확한 피팅 가능

---

## 6. `display: flex` + `align-items: flex-start` (cell-keyword)

**선택**: flex-column + flex-start  
**이유**:
- `.kw-line`(자식)이 부모 폭을 무조건 점유하면 `offsetWidth`로 실제 텍스트 폭 측정 불가
- `align-items: flex-start`로 cross-axis 기본 stretch를 해제 → 각 line이 콘텐츠 폭만큼만 차지

**비유**: 옷장(flex-column)에 옷걸이들이 자기 크기만큼만 걸려있게 하는 것.

## 7. 색상 배치 점수 (Blue noise)

**선택**: 거리별 차등 페널티 + 간격 표준편차 + 짝수/홀수 균형  
**이유**:
- 단순 랜덤 셔플은 동색 인접 클러스터 생성 가능
- 수학적 "Blue noise" 개념: 같은 색은 가까이 두지 않음
- 가중치가 큰 룰: 인접(+80) → 2거리(+35) → 3거리(+12) → 4거리(+4)
- 간격 표준편차 최소화로 강조색이 화면에 균등 분산

---

## 8. Orange 첫 4장 안 노출 (Above the fold)

**선택**: `firstOrange > 3` 이면 큰 페널티  
**이유**:
- 사용자 "유형별이사/실력의차이 탭이 첫 화면에 너무 밋밋"
- 포인트컬러(주홍)가 초기 뷰포트에 보이도록 강제 (+30 per position 초과)
- 300회 시드 셔플 중 이 룰을 만족하는 배치를 선호

---

## 9. 유형별이사/실력의차이 중복 제거

**선택**: 2개 이상 비-하이라이트 태그 후기 → 이 두 탭에서 제외  
**이유**:
- 모든 탭에 중복 노출되면 반복 피로감
- 이 두 탭은 "고유한 카테고리 후기"로 정제
- 나머지 탭(재이사맛집/친절응대 등)은 중복 허용 + v-blue로 시각적 마커

## 10. 배너 중앙 정렬 + 새 카피

**선택**: `text-align: center` + 4줄 강조 문장 구조  
**이유**:
- 사용자 피드백: "실제 고객님들께서 직접 남긴 이야기 / 다음에도 꼭 여기서 / 직접 경험해보세요"
- 중앙 정렬은 강한 메시지 톤에 어울림
- 모바일 14.5px / PC 16px — 타이틀(hero) 너무 크면 배너가 묻히므로 조심스럽게

## 11. 후기 본문 축약 → CSV 원본 복구로 전환 (v04w28)

**선택**: 카드 시트에 뜨는 `fullTextMap` 본문을 **CSV 원본 기준**으로 대폭 복원 (총 74건, +13,000자)  
**이유**:
- 초기 html은 모바일 가독성을 위해 많은 후기를 CSV 대비 심하게 축약 (제니퍼는 원본 1,707자 → 380자로 축약)
- 알론이 "초코바닐라 후기에 '음료수' 문장이 빠져있다"고 발견하며 조사 시작
- 3-way diff(CSV ↔ html ↔ md) 결과 78건 중 76건에서 누락 확인
- 결정: **"후기의 구체적 디테일 자체가 신뢰의 근거"** — 생생한 에피소드(책장 순서, 강아지 종이, 꼭꼬핀 등)야말로 후기 가치

**이전 시도 & 실패**: 
- "카드 시트가 너무 길면 이탈" 우려로 축약본 유지 검토 → 휴대폰 실물 테스트 결과 스크롤 감이 나쁘지 않음 확인
- 폰트 0.8em 축소 옵션 검토 → 줄 합치기가 더 효과적인 것으로 결론

## 12. 폰트 축소(0.8em) 대신 줄 합치기로 가독성 확보

**선택**: 시트 본문 폰트는 현재 크기 유지, **줄 합치기 규칙(14~22자)**로 가독성 개선  
**이유**:
- 제니퍼(1,811자)로 휴대폰 실물 테스트 → 폰트 축소 없어도 스크롤 부담 적음
- 대신 "방 두 개만 하는 거라, 다행히 거실과 부엌은" 같이 짧은 의미 단위 2개를 한 줄에 합치면 스크롤량 확연히 줄어듦
- 폰트 축소는 "일부 카드만 작은 글씨" → 시각적 일관성 해침
- **`.sheet-body--dense` 클래스는 구현 안 함** (코드 복잡도 회피)

**이전 시도 & 실패**:
- 임계값(800자/1000자/1400자) 검토 → 실물 테스트 후 기능 자체 불필요 판정

## 13. CSV 원본을 골드 스탠다드로, md는 중간 레이어로

**선택**: 복구 소스 피라미드 확립 → `CSV → RAW_v02/md → _edit.md → html`  
**이유**:
- 각 레이어에서 조금씩 내용이 소실되거나 편집됨 — CSV가 가장 원본에 가까움
- `_edit.md`에도 중복 문단 발생(편집 실수)하거나 일부 단락 누락 확인
- 복구 방향은 항상 **CSV → html + md** (역방향 아님)
- 단, 상호명 교정/조사 교정 같이 CSV에 없는 변형은 결과물에 반영 (CSV는 원본 GS 시절 작성)

**예외**:
- "GS이사몰" → "이사짐캐리" 변환은 CSV와 달라도 OK
- 명백한 오타("가진도" → "가전도", "장농" → "장롱") 자연스럽게 다듬기 허용
- 이모티콘/구어체는 CSV 그대로 보존

## 14. md 동기화는 html 복구보다 지연 반영 (분리 작업)

**선택**: 중/대규모 34건 복구 시, **html만 먼저 반영**하고 `_edit.md`는 별도 세션에서 동기화  
**이유**:
- html(fullTextMap) 한 라인이 한 후기 → 라인 단위 단순 교체 가능
- md는 블록 구조(헤더/메타/본문/구분선) → 정규식이 섬세해야 함 (이전 세션에서 버그 경험)
- 알론이 "휴대폰에서 html 결과 먼저 확인" 우선 → 배포 속도 중시
- md 동기화는 검증 후 별도 작업으로 분리, 필요시 `apply_md_v2.py` 방식으로 일괄 처리

**단점**: 한동안 md와 html 내용이 다름 (편집용 md는 낡은 상태). 다음 세션에서 동기화 필수.
